# Результат 3

Измерение jitter при работе FreeRTOS с внешней периферией: MPU6050, FIFO датчика, I2C и конвейер задач.

Был создан конвейер из трех задач:

- `I2CReadTask`: периодическая задача, `period_ms = 10`, читает FIFO MPU6050 через I2C и кладет измерение в очередь.
- `ProcessDataTask`: ожидает данные из очереди и умножает все 7 значений измерения на `200`.
- `UartSendTask`: ожидает обработанные данные из очереди. В текущем варианте не печатает каждое измерение, чтобы UART-логирование не доминировало в измерениях.

Отдельная задача `MonitorTask` раз в 10 секунд выводит статистику.

Важно: в этой конфигурации поле `Jitter` имеет разный физический смысл для разных задач:

- Для `I2CReadTask` это **release jitter**: отклонение фактического старта периодической задачи от теоретического времени запуска.
- Для `ProcessDataTask` и `UartSendTask` это **activation latency**: задержка между готовностью данных на предыдущем этапе конвейера и фактическим запуском следующей задачи.

## Интерпретация

В отличие от предыдущих экспериментов, здесь в систему добавлен внешний источник времени: MPU6050 самостоятельно формирует измерения и кладет их во внутренний FIFO. MCU затем периодически опрашивает FIFO через I2C.

Получается несколько взаимодействующих временных доменов:

- планировщик FreeRTOS;
- тактирование ядра MCU;
- блокирующий HAL I2C-драйвер;
- шина I2C;
- внутренняя частота дискретизации MPU6050;
- FIFO внутри MPU6050;
- очереди FreeRTOS в RAM.

Главный источник возмущения в этом эксперименте - не просто RAM и не просто FreeRTOS, а **блокирующее взаимодействие MCU с внешней периферией, которая живет в своем временном домене**.

Особенно показателен первый интервал:

```bash
fifo last/max:14/224
I2C Max C: 11863 us
Misses: 1
```

Размер одного FIFO-сэмпла равен `14` байт. Значит:

```text
224 / 14 = 16 сэмплов
```

То есть в начале измерения FIFO содержал накопленный пакет данных. Задача I2C стала не просто читать один сэмпл, а разгребать накопленный FIFO. Это увеличило чистое время выполнения `C` до примерно `11.8 ms` и вызвало один deadline miss для периода `10 ms`.

После стабилизации FIFO:

```bash
fifo last/max:14/14
I2C Max C: 1824 us
```

Задача обычно читает один сэмпл, и время выполнения становится значительно меньше.

## Почему Avg C отличается между строками

Строки в отчете - это разные задачи, а не последовательные "батчи" одной задачи:

```text
I2C Avg C     ~= 1140 us
Process Avg C ~= 54 us
UART Avg C    ~= 20 us
```

Это ожидаемо:

- `I2CReadTask` выполняет реальные транзакции с внешней периферией и зависит от состояния FIFO.
- `ProcessDataTask` только умножает 7 значений на `200`.
- `UartSendTask` в текущей конфигурации только потребляет обработанный элемент очереди и обновляет статистику.

Поэтому сравнивать `Avg C` между этими строками как один временной ряд нельзя. Это разные этапы конвейера с разной природой нагрузки.

## Release jitter и execution-time jitter

В этом эксперименте важно разделять два разных эффекта.

`I2CReadTask` является периодической задачей. Поэтому ее поле `Jitter` показывает именно **release jitter**:

```text
J_release = actual_start_time - expected_periodic_start_time
```

В устойчивом режиме он мал:

```bash
I2C | Jitter min/avg/max: 8/8/8 us
```

Это означает, что FreeRTOS достаточно точно запускает периодическую I2C-задачу по расписанию. В момент измерения этого jitter сама I2C-транзакция еще не произошла.

Для `ProcessDataTask` и `UartSendTask` поле `Jitter` имеет другой смысл, потому что эти задачи не периодические, а queue-based. Они запускаются не по фиксированному расписанию, а после появления данных в очереди. Поэтому здесь корректнее говорить не `release jitter`, а **activation latency**:

```text
J_process = process_start_time - raw_sample_ready_time
J_uart    = uart_start_time - processed_sample_ready_time
```

Именно поэтому значения для Process и UART могут быть больше, чем release jitter I2C:

```bash
Process | Jitter min/avg/max: 63/63/63 us
UART    | Jitter min/avg/max: 98/100/798 us
```

Это задержка реакции конвейера после готовности данных: пробуждение задачи из очереди, решение планировщика, context switch и конкуренция с другими ready-задачами.

Основное влияние I2C проявляется не в release jitter, а в **execution-time jitter** задачи I2C. Его можно оценивать как разброс чистого времени выполнения `C`:

```text
J_C = C_max - C_min
```

Например:

```bash
I2C | Max C: 1824 us | Min C: 237 us | Avg C: 1140 us
```

Тогда:

```text
J_C = 1824 - 237 = 1587 us
```

Это означает, что время выполнения I2C-задачи существенно меняется от запуска к запуску. Причина - блокирующие I2C-транзакции и зависимость от состояния внешнего устройства: сколько данных накоплено в FIFO, в какой фазе находится датчик, сколько байт нужно считать и сколько времени займет обмен по шине.

Кратко:

- release jitter I2C мал, потому что периодическую задачу планировщику легко запускать стабильно;
- activation latency Process/UART больше, потому что эти задачи запускаются реактивно через очереди;
- execution-time jitter I2C значительно больше, потому что задача зависит от блокирующего обмена с внешней периферией.

## Вывод

CPU-only эксперименты показывали, что FreeRTOS достаточно стабильно выдерживает release jitter в простых периодических задачах. Однако добавление MPU6050 через I2C резко меняет картину:

- время выполнения I2C-задачи становится зависимым от состояния FIFO;
- внешний датчик производит данные независимо от планировщика MCU;
- блокирующий HAL I2C-драйвер удерживает CPU во время обмена;
- downstream-задачи получают не release jitter, а задержку активации после готовности данных;
- накопление FIFO может создавать burst-нагрузку и deadline misses.

Формулировка для диплома:

> При добавлении внешней периферии через I2C источник временной неопределенности переносится из чисто вычислительной области в область взаимодействия нескольких независимых временных доменов. MPU6050 формирует данные независимо от FreeRTOS, FIFO сглаживает, но также может накапливать burst-нагрузку, а блокирующий I2C-драйвер превращает состояние периферии в переменное время выполнения задачи. Поэтому наблюдается рост response time и execution-time variability даже при сравнительно стабильном release jitter планировщика.
